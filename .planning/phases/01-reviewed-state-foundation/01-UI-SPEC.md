## Spacing Scale

Tailwind 3.4 default scale (multiples of 4px). Phase 1 declares the following subset for new chrome:

| Token | Value | Usage in Phase 1 |
|-------|-------|------------------|
| `p-0.5` / `p-1` | 2-4px | Icon button inner padding (matches existing favorite Star at `EventCard.tsx:131-148`) |
| `gap-1` / `gap-2` | 4-8px | Action-cluster gaps in the EventCard top-right (reviewed button next to Star) |
| `p-2` / `gap-2` | 8px | Selection checkbox overlay padding; bulk action bar inner gap |
| `p-3` / `px-4` | 12-16px | Bulk action bar horizontal padding; row-action slot in NotificationHistory |
| `gap-3` / `gap-4` | 12-16px | Bulk action bar group gaps (count → select-all → primary → cancel) |
| `py-2` / `py-3` | 8-12px | Bulk action bar vertical padding (thumb-reachable height ≥ 48px on mobile) |
| `mb-2` / `mt-2` | 8px | Reviewed badge offset from EventDetail title baseline |
| Bottom inset | 64-72px | EventList scroll padding when bulk action bar is active (avoids Pitfall 8 — bar covering last cards) |

**Hit-target floor:** Every new tap target is ≥ 44x44 logical px on touch (matches AGENTS.md mobile expectations + Apple HIG / Android Material). On EventCard the existing favorite Star already uses `p-1` inside an `h-9 w-9`-equivalent button; the reviewed button matches it exactly so the cluster keeps a consistent rhythm.

**Exceptions:** None. Reviewed badge corner offset uses Tailwind `top-2 right-2` (8px) for consistency with existing card-corner overlays.

**Accent reserved for (explicit list, never widened in this phase):**

### Reviewed-state visual treatment matrix

| State | Card opacity | Icon | Badge | Aria-pressed | Test ID resting → active |
|-------|-------------|------|-------|--------------|--------------------------|

## Toggle Interaction Contract

### EventCard toggle (`event-reviewed-button`)

| Property | Value |
|----------|-------|
| Element | `<button>` with `data-testid="event-reviewed-button"` |
| Hit area | `h-9 w-9` (≥36px); inside the existing top-right action cluster, padding `p-1` around an `h-5 w-5` icon → effectively 44x44 on touch |

## Bulk Action Feedback Contract

| Step | UI behavior | Toast |
|------|-------------|-------|
| User taps "Mark reviewed" with N items selected | Bar disables primary button briefly (`disabled` attribute for ~50ms) — purely a debounce; the operation is sync since it's a single Zustand `set` (Pitfall 3 mitigation) | — |
| Optimistic update | All N cards immediately render reviewed: `opacity-60` + corner check-mark badge; their reviewed icon-button switches to active state | — |
| Selection cleared | Bar slides out; per-card checkboxes disappear; `ring-primary/40` rings disappear | — |
| Success toast | `sonner` `toast.success(t('events.bulk.marked_toast', { count: N }))` with default duration (4s) | "Marked {{count}} events reviewed" |
| Cross-store cascade | Matching notification rows in current profile flip to `read` via `markEventRead` (D-15/D-16) — invisible from the events list, visible if user navigates to NotificationHistory | — |
| Failure (cap-overflow) | No user-facing error. `log.profile` writes INFO with overflow count. The drop-lowest-Ids semantics is correct because cutoff covers them implicitly. | — |

**No skeleton / no spinner:** Every operation is synchronous local state. Showing a spinner would imply network latency that doesn't exist.

## Profile-Switch Visual Behavior

| Concern | Behavior |
|---------|----------|
| Indicator state on switch | Re-derives instantly via `useReviewedState` re-binding to new `profileId` from `useCurrentProfile`. No animation, no loading state, no flicker — Zustand selector identity is stable through the switch and React only re-renders the cards whose state actually changed. |
| Cutoff state on switch | If the new profile has no cutoff yet, all events render unreviewed AND the `NotificationBadge` suppresses count (D-02). When the user lands on the Events page, the seeder fires on first React Query success. |
| Selection-mode state on switch | Selection is component-local (NOT persisted) → automatically clears when `Events.tsx` unmounts during the switch's `clearQueryCache` cycle. |
| Bulk action bar on switch | If a switch happens mid-selection (rare; switch is usually triggered from Profiles page), the bar disappears with the unmount. No special handling. |
| Loading skeleton | Not added. The existing events-loading skeleton (already in `Events.tsx` / `EventListView`) is reused as-is. Reviewed state is a passive overlay, not a primary content load. |
| Stale indicator risk | Zero — data is keyed by `profileId` inside the persisted store; no cross-profile leak by construction. Verified by the mandatory unit test `does not affect other profiles` (RESEARCH.md §Profile-Scoping). |

## Open Questions Forwarded to Planner / Checker

1. **Toggle on cutoff-implied reviewed event** (RESEARCH.md A1 / Open Question 1) — Phase 1 default: tap is a no-op for events whose `Id <= cutoff` and not in explicit set. Confirm before implementation; alternative is to add a `profileExceptions` field (deferred to v2 per recommendation).
2. **`EventReviewedBadge.tsx` extraction** (RESEARCH.md A2 / Open Question 3) — Recommended; saves duplication between EventCard and NotificationHistory rows. Planner may collapse into inline render if preferred; both are acceptable.
3. **Range-select on tablet/desktop with shift-click** — Locked Phase 1 default is plain tap-toggle in selection mode. Range-select is a discretionary enhancement; if planner adds it, confirm `data-testid` and update the test matrix.
4. **EventDetail "active" toggle visual** — Two acceptable patterns: (a) Button shows `t('events.reviewed.unmark')` when reviewed (active toggle metaphor) or (b) Button shows `t('events.reviewed.label')` ("Reviewed") with a check (status metaphor). Both translation keys ship; planner picks the visible one. Recommendation: **(b)** — the static label "Reviewed" is calmer and the leading green check carries the toggle affordance via `aria-pressed`.
